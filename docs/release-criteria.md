# Develop -> Production 승격 기준

이 문서는 develop 브랜치에서 production 브랜치로 승격할 때의 공통 기준과 절차를 정의합니다.

## 1. 승격 전 필수 기준

### 기능 기준
- 릴리스 범위가 명확하게 정의되어 있다.
- 사용자 영향 변경(회원가입/로그인/타이머/데이터 저장)의 수동 점검이 끝났다.
- DB/파일 포맷 변경이 있으면 호환성 확인 또는 마이그레이션 절차가 문서화되어 있다.

### 품질 기준
- develop 브랜치 CI(테스트 워크플로우) 성공.
- 로컬 기본 점검 통과:
  - npm test
  - 핵심 흐름 수동 점검(로그인, 타이머 시작/정지, 데이터 저장)
- 콘솔 에러/네트워크 5xx가 없는 상태.

### 보안/운영 기준
- 민감정보(.env, 키, 토큰)가 커밋되지 않았다.
- 운영 배포용 GitHub Secrets 설정 완료:
  - AZURE_WEBAPP_NAME
  - AZURE_WEBAPP_PUBLISH_PROFILE
- 롤백 방법(이전 커밋 또는 이전 릴리스 태그)이 준비되어 있다.

### 승인 기준
- 최소 1명 이상 리뷰 승인.
- 운영 반영 시간/담당자/검증 담당자 지정 완료.

## 2. 권장 승격 절차

1. develop 최신화 후 릴리스 브랜치(선택) 또는 PR 생성
- develop -> production PR 생성
- 변경 요약, 테스트 증빙, 위험도, 롤백 계획 작성

2. 리뷰 및 최종 확인
- 리뷰 승인 획득
- 체크리스트 전 항목 확인

3. 병합 전략
- 기본: Squash merge 권장(릴리스 단위 추적 쉬움)
- 병합 메시지: feat/release 성격이 드러나게 작성

4. 배포
- production 브랜치 반영 후 GitHub Actions 배포 워크플로우 실행 확인
- Azure Web App 반영 확인(헬스체크, 핵심 사용자 시나리오)

5. 사후 검증
- 배포 후 10~30분 관찰
- 에러율, 로그인/핵심 기능 정상 여부 확인

## 3. 롤백 기준

아래 중 하나라도 발생하면 즉시 롤백 검토:
- 로그인/회원가입 불가
- 타이머 핵심 기능 동작 불가
- 서버 5xx 급증
- 데이터 손실/불일치 의심

롤백 방법:
- production에서 이전 정상 커밋으로 revert PR 생성 후 즉시 반영
- 또는 사전에 만든 릴리스 태그 기준으로 복원

## 4. 브랜치 운영 원칙

- 개발 작업: develop에서만 진행
- 운영 배포: production으로 병합된 코드만 허용
- 긴급 수정: hotfix 브랜치 생성 -> production 우선 반영 -> develop 역병합
